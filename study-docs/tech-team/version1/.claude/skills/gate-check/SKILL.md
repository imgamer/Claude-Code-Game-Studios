# Phase Gate Validation (UE 5.8, TD-only)

This skill validates whether the project is ready to advance to the next
development phase. It checks for required artifacts, quality standards, and
blockers.

**Distinct from `/project-stage-detect`**: That skill is diagnostic ("where are
we?"). This skill is prescriptive ("are we ready to advance?" with a formal
verdict).

**Director panel (simplified):** The original project spawned four parallel
director gates (CD/PR/AD/TD). This team is a technical implementation team —
creative, producer, art, and narrative director gates do not apply. Only
**`TD-PHASE-GATE`** (technical-director) is spawned here.

---

## Production Stages (3)

The project progresses through these 3 stages (compressed from the original 7
— Concept/Systems Design/Pre-Production/Polish/Release are removed; this team
receives feature specs directly from the user):

1. **Technical Setup** — Engine config (UE 5.8 pinned), architecture decisions,
   control manifest, architecture review
2. **Requirements Breakdown** — Epics and stories decomposed from feature specs
3. **Implementation** — Feature development (story loop: readiness → dev-story →
   code-review → story-done; optional `/team-feature` for complex parallel
   features)

**When a gate passes**, write the new stage name to `production/stage.txt`
(single line, e.g. `Implementation`). This updates the status line immediately.

## 1. Parse Arguments

**Target phase:** `$ARGUMENTS[0]` (blank = auto-detect current stage, then
validate the next transition)

Also resolve the review mode (once, store for all gate spawns this run):
1. If `--review [full|lean|solo]` was passed → use that
2. Else read `production/review-mode.txt` → use that value
3. Else → default to `lean`

Note: in `solo` mode, the TD-PHASE-GATE spawn is skipped — gate-check becomes
artifact-existence checks only. In `lean` mode, TD-PHASE-GATE still runs (phase
gates are the purpose of lean mode). In `full` mode, TD-PHASE-GATE runs as
normal.

- **With argument**: `/gate-check implementation` — validate readiness for that
  specific phase
- **No argument**: Auto-detect current stage using the same heuristics as
  `/project-stage-detect`, then **confirm with the user before running**:

  Use `AskUserQuestion`:
  - Prompt: "Detected stage: **[current stage]**. Running gate for [Current] →
    [Next] transition. Is this correct?"
  - Options:
    - `[A] Yes — run this gate`
    - `[B] No — pick a different gate` (if selected, show a second widget
      listing all gate options: Technical Setup → Requirements Breakdown,
      Requirements Breakdown → Implementation)

  Do not skip this confirmation step when no argument is provided.

## 2. Phase Gate Definitions

### Gate: (start) → Technical Setup

This is the entry gate — usually auto-passed once `/setup-engine` has been run.
Validate only that the engine is configured.

**Required Artifacts:**
- [ ] `CLAUDE.md` Technology Stack is set to Unreal Engine 5.8 (not `[CHOOSE]`)
- [ ] `.claude/docs/technical-preferences.md` exists and is populated (naming
      conventions, performance budgets, target platforms, test framework)
- [ ] `.claude/docs/engine-specialists.md` exists (UE sub-specialist routing
      table written by `/setup-engine`)

**Quality Checks:**
- [ ] Technical preferences have naming conventions and performance budgets set
- [ ] UE 5.8 Knowledge Authority section present in `technical-preferences.md`

---

### Gate: Technical Setup → Requirements Breakdown

**Required Artifacts:**
- [ ] Master architecture document exists at `docs/architecture/architecture.md`
- [ ] At least 3 Architecture Decision Records in `docs/architecture/` covering
      Foundation-layer systems (scene management, event architecture, input,
      save/load as applicable)
- [ ] Control manifest exists at `docs/architecture/control-manifest.md`
      (generated by `/create-control-manifest` from Accepted ADRs)
- [ ] `/architecture-review` has been run (a review report file exists in
      `docs/architecture/`)
- [ ] Architecture traceability index exists at
      `docs/architecture/requirements-traceability.md`
- [ ] `docs/architecture/tr-registry.yaml` exists and indexes at least the
      feature specs in `docs/requirements/`
- [ ] At least one feature spec exists under `docs/requirements/`

**Quality Checks:**
- [ ] All Foundation-layer ADRs have status `Accepted` (not `Proposed`) —
      stories cannot be unblocked until their governing ADR is accepted
- [ ] All ADRs have an **Engine Compatibility section** stamped with
      `Unreal Engine 5.8` and Context7 verification notes
- [ ] All ADRs have a **Requirements Addressed section** with explicit feature
      spec / TR-ID linkage (not GDD linkage — this team has no GDDs)
- [ ] No ADR references UE APIs that Context7 flagged as deprecated in 5.8
- [ ] Architecture traceability matrix has **zero Foundation layer gaps**
      (all Foundation requirements must have ADR coverage before Requirements
      Breakdown)

**ADR Circular Dependency Check**: For all ADRs in `docs/architecture/`, read
each ADR's "ADR Dependencies" / "Depends On" section. Build a dependency graph
(ADR-A → ADR-B means A depends on B). If any cycle is detected (e.g. A→B→A, or
A→B→C→A):
- Flag as **FAIL**: "Circular ADR dependency: [ADR-X] → [ADR-Y] → [ADR-X].
  Neither can reach Accepted while the cycle exists. Remove one 'Depends On'
  edge to break the cycle."

---

### Gate: Requirements Breakdown → Implementation

**Required Artifacts:**
- [ ] Epics defined in `production/epics/` with at least Foundation and Core
      layer epics present (use `/create-epics layer: foundation` then
      `/create-epics layer: core`)
- [ ] Stories exist for each Foundation-layer epic (use
      `/create-stories [epic-slug]`)
- [ ] At least one sprint plan or active-story record exists under
      `production/sprints/` or `production/session-state/`
- [ ] All Foundation and Core layer ADRs remain `Accepted` (not regressed to
      Proposed)
- [ ] Control manifest remains in sync with Accepted ADRs

**Quality Checks:**
- [ ] Sprint plan / active stories reference real story file paths from
      `production/epics/` (stories must embed TR-ID + ADR reference, not just
      feature spec names)
- [ ] Stories have passed `/story-readiness` before being picked up (at least
      Foundation-layer stories)
- [ ] Architecture document has no unresolved open questions in Foundation or
      Core layers
- [ ] Manual validation confirms feature specs + architecture + epics are
      coherent (run `/architecture-review` if not done recently)
- [ ] No stale ADR Engine Compatibility sections (all stamped with UE 5.8 and
      Context7-verified APIs)

## 3. Run the Gate Check

**Before running artifact checks**, read `docs/consistency-failures.md` if it
exists. Extract entries whose Domain matches the target phase. Carry these as
context — recurring conflict patterns in the target domain warrant increased
scrutiny on those specific checks.

For each item in the target gate:

### Artifact Checks
- Use `Glob` and `Read` to verify files exist and have meaningful content
- Don't just check existence — verify the file has real content (not just a
  template header)
- For code checks, verify directory structure and file counts under `Source/`

### Quality Checks
- For test checks: Run the test suite via `Bash` if a test runner is configured
- For ADR checks: `Read` each ADR and verify the Engine Compatibility section
  is stamped with UE 5.8 and Context7 verification notes
- For performance checks: `Read` technical-preferences.md and compare against
  any profiling data in `tests/performance/`
- For localization checks: `Grep` for hardcoded strings in `Source/`

### Cross-Reference Checks
- Compare `docs/requirements/` feature specs against `Source/` implementations
- Check that every system referenced in architecture docs has corresponding
  code (once in Implementation)
- Verify sprint plans reference real story file paths

## 4. Collaborative Assessment

For items that can't be automatically verified, **ask the user**:

- "I can't automatically verify that the feature spec is internally coherent.
  Has it been reviewed?"
- "Performance profiling data isn't available. Would you like to defer this
  check until Implementation?"
- "No smoke check report found yet — acceptable to defer until first sprint
  close-out?"

**Never assume PASS for unverifiable items.** Mark them as MANUAL CHECK NEEDED.

---

## 4b. Technical Director Assessment (TD-PHASE-GATE only)

**Apply review mode before spawning:**
- `solo` → skip TD-PHASE-GATE. Note in output: "TD-PHASE-GATE skipped — Solo
  mode. Gate verdict based on artifact and quality checks only." Proceed to
  Phase 5.
- `lean` → spawn TD-PHASE-GATE (phase gates always run in lean mode — this is
  their purpose).
- `full` → spawn TD-PHASE-GATE as normal.

(Review mode was resolved in Phase 1. Use that stored value here.)

Before generating the final verdict, spawn **`technical-director`** via Task
using gate **TD-PHASE-GATE** (`.claude/docs/director-gates.md`).

Pass to the director: target phase name, list of artifacts present, and the
context fields listed in that gate's definition.

**Present the director's verdict:**

```
## Technical Director Assessment

Technical Director: [READY / CONCERNS / NOT READY]
  [feedback]
```

> Note: The original project spawned four directors in parallel (CD/PR/AD/TD)
> and took the strictest verdict. This team spawns only the technical-director
> — creative, producer, art, and narrative gates do not apply to a technical
> implementation team.

**Apply to the verdict:**
- TD returns NOT READY → verdict is minimum FAIL (user may override with
  explicit acknowledgement)
- TD returns CONCERNS → verdict is minimum CONCERNS
- TD returns READY → eligible for PASS (still subject to artifact and quality
  checks from Section 3)

## 5. Output the Verdict

```
## Gate Check: [Current Phase] → [Target Phase]

**Date**: [date]
**Checked by**: gate-check skill (TD-only)

### Required Artifacts: [X/Y present]
- [x] docs/architecture/architecture.md — exists, 4.2KB
- [ ] docs/architecture/control-manifest.md — MISSING
- [x] docs/requirements/ — 2 feature specs present

### Quality Checks: [X/Y passing]
- [x] All Foundation ADRs status Accepted
- [ ] ADR-0002 missing Engine Compatibility section stamped with UE 5.8
- [?] Feature spec internal coherence — MANUAL CHECK NEEDED

### Blockers
1. **No control manifest** — Run `/create-control-manifest` to generate it
   from your Accepted ADRs before advancing.
2. **ADR-0002 missing UE 5.8 Engine Compatibility section** — Re-run
   `/architecture-decision retrofit docs/architecture/adr-0002-*.md` or
   manually add the section.

### Recommendations
- [Priority actions to resolve blockers]
- [Optional improvements that aren't blocking]

### Verdict: [PASS / CONCERNS / FAIL]
- **PASS**: All required artifacts present, all quality checks passing, TD READY
- **CONCERNS**: Minor gaps exist but can be addressed during the next phase
- **FAIL**: Critical blockers must be resolved before advancing
```

## 5a. Chain-of-Verification

After drafting the verdict in Phase 5, challenge it before finalising.

**Step 1 — Generate 5 challenge questions** designed to disprove the verdict:

> **Tool-action requirement**: At least 2 of the 5 challenge questions below
> must be answered by re-reading a specific file (Read tool) or re-running a
> specific check (Grep tool) — not by reflection alone. Mark these with
> [TOOL ACTION] to indicate a tool was used.

For a **PASS** draft:
- "Which quality checks did I verify by actually reading a file, vs. inferring
  they passed?"
- "Are there MANUAL CHECK NEEDED items I marked PASS without user
  confirmation? [TOOL ACTION] Re-scan the checklist for any [?] or MANUAL CHECK
  items."
- "Did I confirm all listed artifacts have real content, not just empty
  headers? [TOOL ACTION] Re-read the file and check it has non-placeholder
  content."
- "Could any blocker I dismissed as minor actually prevent the phase from
  succeeding?"
- "Which single check am I least confident in, and why?"

For a **CONCERNS** draft:
- "Could any listed CONCERN be elevated to a blocker given the project's
  current state?"
- "Is the concern resolvable within the next phase, or does it compound over
  time?"
- "Did I soften any FAIL condition into a CONCERN to avoid a harder verdict?"
- "Are there artifacts I didn't check that could reveal additional blockers?"
- "Do all the CONCERNS together create a blocking problem even if each is
  minor alone?"

For a **FAIL** draft:
- "Have I accurately separated hard blockers from strong recommendations?"
- "Are there any PASS items I was too lenient about?"
- "Am I missing any additional blockers the user should know about?"
- "Can I provide a minimal path to PASS — the specific 3 things that must
  change?"
- "Is the fail condition resolvable, or does it indicate a deeper design
  problem?"

**Step 2 — Answer each question** independently.
Do NOT reference the draft verdict text — re-check specific files or ask the
user.

**Step 3 — Revise if needed:**
- If any answer reveals a missed blocker → upgrade verdict (PASS→CONCERNS or
  CONCERNS→FAIL)
- If any answer reveals an over-stated blocker → downgrade only if citing
  specific evidence
- If answers are consistent → confirm verdict unchanged

**Step 4 — Note the verification** in the final report output:
`Chain-of-Verification: [N] questions checked — verdict [unchanged | revised
from X to Y]`

## 6. Update Stage on PASS

When the verdict is **PASS** and the user confirms they want to advance:

1. Write the new stage name to `production/stage.txt` (single line, no trailing
   newline)
2. This immediately updates the status line for all future sessions

Example: if passing the "Requirements Breakdown → Implementation" gate:
```bash
echo -n "Implementation" > production/stage.txt
```

**Always ask before writing**: "Gate passed. May I update `production/stage.txt`
to 'Implementation'?"

## 7. Closing Next-Step Widget

After the verdict is presented and any stage.txt update is complete, close with
a structured next-step prompt using `AskUserQuestion`.

**Tailor the options to the gate that just ran:**

For **Technical Setup PASS** (→ Requirements Breakdown):
```
Gate passed. What would you like to do next?
[A] Run /create-control-manifest — generate the layer rules manifest from your
    Accepted ADRs (required before epics)
[B] Write more ADRs first — run /architecture-decision [next-system]
[C] Run /create-epics layer:foundation — start decomposing feature specs into
    epics (requires control manifest)
[D] Stop here for this session
```

> **Note for Technical Setup PASS**: `/create-control-manifest` is the required
> next step before writing any epics. It extracts the technical rules from
> Accepted ADRs that stories will be checked against.

For **Requirements Breakdown PASS** (→ Implementation):
```
Gate passed. What would you like to do next?
[A] Run /story-readiness [story-path] — validate the first story before
    implementation
[B] Run /dev-story [story-path] — start implementing the first ready story
[C] Run /team-feature [feature-description] — for a complex multi-agent feature
[D] Stop here for this session
```

For all other gates, offer the two most logical next steps for that phase plus
"Stop here".

## 8. Follow-Up Actions

Based on the verdict, suggest specific next steps (only skills that exist in
this team are listed — cut skills are omitted):

- **No technical preferences?** → `/setup-engine` to configure UE 5.8 project
  settings
- **Missing ADRs?** → `/architecture-decision [topic]` for individual decisions
- **No master architecture doc?** → `/create-architecture` for the full
  blueprint
- **ADRs missing UE 5.8 Engine Compatibility sections?** → Re-run
  `/architecture-decision retrofit [adr-path]` or manually add the section
- **No architecture review report?** → `/architecture-review`
- **Missing control manifest?** → `/create-control-manifest` (requires Accepted
  ADRs)
- **Missing epics?** → `/create-epics layer: foundation` then
  `/create-epics layer: core` (requires control manifest)
- **Missing stories for an epic?** → `/create-stories [epic-slug]`
- **Stories not implementation-ready?** → `/story-readiness` to validate
- **Tests failing?** → delegate to `lead-programmer` or `qa-tester`
- **No QA plan for current sprint?** → `/qa-plan` to generate one
- **Need a smoke check?** → `/smoke-check`

## Collaborative Protocol

This skill follows the collaborative design principle:

1. **Scan first**: Check all artifacts and quality gates
2. **Ask about unknowns**: Don't assume PASS for things you can't verify
3. **Present findings**: Show the full checklist with status
4. **User decides**: The verdict is a recommendation — the user makes the final
   call
5. **Get approval**: "May I write this gate check report to
   `production/gate-checks/`?"
6. **Never auto-fix**: If required artifacts are missing, report the FAIL
   verdict and name the skill to run (e.g. "run `/create-control-manifest`").
   Do NOT create missing files or re-run the gate automatically. Creating files
   to manufacture a PASS defeats the gate's purpose.

**Never** block a user from advancing — the verdict is advisory. Document the
risks and let the user decide whether to proceed despite concerns.
