# 13 · 术语对照表 —— 看懂这个项目的行话

这个项目借用了游戏开发流程的术语 + Claude Code 的术语 + 自创的缩写体系。初学者会被 `TD-ENGINE-RISK`、`ADR Accepted`、`LP-FEASIBILITY`、`TR-COMBAT-003` 这些词砸晕。

这篇文档把所有行话提取出来，按"游戏开发流程 / 智能体编排 / 项目自创缩写"三类分组，逐一解释。

---

## 第一类：游戏开发流程术语

### Stage / Phase（阶段）

游戏开发被本项目切成 **7 个阶段**，每个阶段有明确的进出条件：

| 阶段 | 英文 | 干什么 | 出口门禁 |
|------|------|--------|----------|
| 概念 | Concept | 把游戏点子写成文档（核心支柱、玩法循环） | Concept → Systems Design |
| 系统设计 | Systems Design | 拆成系统，每个系统写一份 GDD | Systems Design → Technical Setup |
| 技术搭建 | Technical Setup | 定架构、写 ADR、搭引擎、配置测试 | Technical Setup → Pre-Production |
| 预制作 | Pre-Production | 写故事卡、跑冲刺、验证可行性 | Pre-Production → Production |
| 制作 | Production | 大规模实现功能 | Production → Polish |
| 打磨 | Polish | 修 bug、调手感、优化性能 | Polish → Release |
| 发布 | Release | 上线 | — |

**关键概念**：阶段之间**不能直接跳**，必须过"门禁"（见下文）。这是游戏工业的标准流水线，源自 [workflow-catalog.yaml](file:///workspace/.claude/docs/workflow-catalog.yaml)。

---

### GDD — Game Design Document（游戏设计文档）

**是什么**：描述一个游戏系统"该怎么玩"的文档。一个系统一份 GDD。

**放哪**：[design/gdd/](file:///workspace/design/gdd/)，比如 `combat.md`（战斗系统）、`inventory.md`（背包系统）。

**包含什么**：机制概述、玩家幻想、详细规则、公式、边缘情况、依赖、调参旋钮、验收标准。

**类比**：盖楼前的"设计图纸"——说清楚这栋楼要几个房间、每间多大、什么用途，但不涉及用几号钢筋。

---

### ADR — Architecture Decision Record（架构决策记录）

**是什么**：记录一个架构决策的文档——**为什么选 A 不选 B**。

**放哪**：`docs/architecture/adr-NNNN-标题.md`，比如 `adr-0003-melee-combat.md`。

**4 种状态**（重要！）：

| 状态 | 含义 | 能不能据此实现代码 |
|------|------|---------------------|
| `Proposed`（已提议） | 还在讨论，没定下来 | **不能**。`/dev-story` 会拒绝实现 |
| `Accepted`（已接受） | 定稿了，可以执行 | 能 |
| `Deprecated`（已废弃） | 不再使用 | 不能 |
| `Superseded by ADR-NNNN`（被取代） | 被另一个 ADR 替换 | 看新 ADR |

**"ADR Approve" 的意思**：决策经过评审，状态从 `Proposed` → `Accepted`。只有 `Accepted` 的 ADR 才能指导编码。

**类比**：工程会议纪要——"我们决定用 Godot 而不是 Unity，原因是 X、Y、Z"。后人翻出来就知道为什么这么选的。

**为什么重要**：本项目 `/dev-story` 在派程序员前会先查 ADR 状态。如果是 `Proposed`，直接拒绝实现，让你先跑 `/architecture-decision` 把它定下来。这防止"基于没定稿的决策写代码"。

---

### TR — Technical Requirement（技术需求）

**是什么**：从 GDD 里提炼出的"架构必须满足什么"的技术需求条目。

**编号格式**：`TR-[系统]-NNN`，比如 `TR-COMBAT-003`（战斗系统的第 3 条技术需求）。

**放哪**：登记在 [docs/architecture/tr-registry.yaml](file:///workspace/docs/architecture/tr-registry.yaml)。

**和 GDD/ADR 的关系**：

```
GDD（设计文档）
  ↓ 提炼出
TR（技术需求）—— 比如"命中检测必须 < 1 帧"
  ↓ 被满足于
ADR（架构决策）—— 比如"用 Godot Area3D 做命中检测"
  ↓ 被实现于
Story（故事卡）—— 比如"实现近战命中盒"
```

**类比**：GDD 说"战斗要爽"，TR 把它翻译成"命中检测延迟 < 16ms"这种可衡量的技术指标，ADR 决定怎么实现，Story 是具体任务卡。

---

### Story / Epic（故事卡 / 史诗）

**Story（故事卡）**：最小可执行的任务单元。一张卡 = 一个程序员一两天能干完的活。

**Epic（史诗）**：一组相关 Story 的集合，规模更大。

**放哪**：[production/epics/](file:///workspace/production/epics/)，比如 `production/epics/combat/melee-hitbox.md`。

**包含什么**：标题、ID、层（Layer）、类型（Type）、关联的 TR-ID、关联的 ADR、验收标准、依赖、Out of Scope（明确不做的事）。

**类比**：敏捷开发里的 User Story。一张卡问"作为 X，我想 Y，以便 Z"。

---

### Pillar（支柱）

**是什么**：游戏的 3-5 条核心体验承诺。所有设计决策都要服从支柱。

**举例**：
- "每一场战斗都必须紧张刺激"
- "玩家始终感觉资源稀缺"
- "世界对玩家选择有反应"

**用途**：`CD-PILLARS` 门禁就是拿设计文档去"压测"支柱——你的设计是否违背了你自己定的核心承诺？

**类比**：公司价值观。任何决策都要和价值观对齐。

## 第二类：智能体编排术语

### Agent（智能体）

**是什么**：一个用 `.md` 文件定义的"专家角色"。Claude Code 通过内置 `Agent`/`Task` 工具派生它。

**放哪**：`.claude/agents/[name].md`（项目级）或 `~/.claude/agents/[name].md`（全局级）。

**本项目 49 个 agent**：比如 `gameplay-programmer`（玩法程序员）、`creative-director`（创意总监）、`qa-tester`（QA 测试员）。

详见上一篇 [12-从零编排智能体协作](file:///workspace/study-docs/12-从零编排智能体协作.md)。

---

### Subagent（子智能体）

**是什么**：被 `Agent`/`Task` 工具派生出来的智能体实例。它在**独立上下文**里干活，干完返回一段文本给主会话。

**关键特性**：父子之间只有一根数据通道——`prompt` 字符串。子智能体看不到主会话历史，得自己读文件。

---

### Skill（技能）

**是什么**：一段可复用的"作业指导书"，定义主会话在某个任务里该派谁、按什么顺序、怎么决策。

**放哪**：`.claude/skills/[name]/SKILL.md`。

**怎么触发**：敲 `/技能名`，比如 `/dev-story`、`/team-combat`、`/gate-check`。

**和 Agent 的区别**：Agent 是"角色档案"（谁），Skill 是"流程编排"（怎么干）。Skill 可以派多个 Agent。

---

### Hook（钩子）

**是什么**：在特定事件自动触发的脚本。比如每次派子智能体时自动记日志。

**放哪**：[.claude/hooks/](file:///workspace/.claude/hooks/)，配置在 [.claude/settings.json](file:///workspace/.claude/settings.json)。

**常用事件**：`PreToolUse`（工具调用前）、`PostToolUse`（工具调用后）、`SubagentStart`（子智能体启动）、`SubagentStop`（子智能体停止）。

---

### Task / Agent 工具

**是什么**：Claude Code **内置工具**，用来派生子智能体。v2.1.63 前叫 `Task`，之后叫 `Agent`，`Task` 仍作别名可用。

**关键澄清**：这个工具**不是用户定义的**。用户定义的是 agent（`.md` 文件），工具是自带的。

## 第三类：项目自创缩写（门禁 ID 体系）

这是最容易让初学者懵的部分。项目用 `XX-YYYY` 格式给每个"门禁"编了号。**前面两个字母 = 谁的门禁**，**后面 = 这个门禁查什么**。

### 门禁前缀对照表

| 前缀 | 全称 | 是谁 | 什么领域 |
|------|------|------|----------|
| `CD-` | Creative Director | 创意总监 | 游戏愿景、支柱、叙事、玩家体验 |
| `TD-` | Technical Director | 技术总监 | 架构、引擎、可行性、技术风险 |
| `PR-` | Producer | 制作人 | 范围、时间线、冲刺、里程碑 |
| `AD-` | Art Director | 美术总监 | 视觉风格、美术圣经 |
| `LP-` | Lead Programmer | 主程序员 | 实现可行性、代码审查 |
| `QL-` | QA Lead | QA 主管 | 故事就绪度、测试覆盖 |
| `ND-` | Narrative Director | 叙事总监 | 叙事一致性 |

**记忆法**：
- `C/T/P/A` 开头的是 **4 大总监**（Tier 1），管宏观
- `L/Q/N` 开头的是 **Lead 主管**（Tier 2），管执行
- `-PHASE-GATE` 结尾的是 **阶段切换门禁**，最关键

---

### 门禁（Gate）是什么

**门禁 = 一道质量检查关卡**。在某个动作之前或阶段切换时，派一个总监 agent 去审一遍，给出 APPROVE / CONCERNS / REJECT 的裁决。

**三态裁决**（来自 [director-gates.md](file:///workspace/.claude/docs/director-gates.md)）：

| 裁决 | 含义 | 后续动作 |
|------|------|----------|
| `APPROVE` / `READY` | 通过 | 继续下一步 |
| `CONCERNS` | 有顾虑但可推进 | 记录顾虑，继续，后续解决 |
| `REJECT` / `NOT READY` | 不通过 | 必须解决问题才能推进 |

**门禁的两种触发方式**：

1. **阶段门禁**（PHASE-GATE）：阶段切换时由 `/gate-check` 并行派 4 个总监，每个跑自己的 `-PHASE-GATE`。一个 NOT READY 全部 FAIL。
2. **技能内门禁**：某个 skill 执行中派特定总监跑特定门禁。比如 `/architecture-decision` 写完 ADR 后跑 `TD-ADR` 审一遍。

### TD-ENGINE-RISK 详解（你问的这个）

**全名**：Engine Version Risk Review（引擎版本风险审查）

**谁跑**：`technical-director` agent

**什么时候触发**（来自 [director-gates.md#L371](file:///workspace/.claude/docs/director-gates.md)）：
> When making architecture decisions that touch post-cutoff engine APIs, or before finalizing any engine-specific implementation approach

翻译：当架构决策用到**超出 LLM 知识截止日期**的引擎 API，或定稿任何引擎特定实现方案之前。

**为什么需要这个门禁**：LLM 的训练数据有截止日期。如果 Godot 4.4 在 LLM 截止后改了某 API 签名，LLM 会"自信地"写出过时代码。这个门禁让技术总监去查 `docs/engine-reference/[engine]/VERSION.md`，核对 API 是否还存在、签名是否变了。

**审什么**：
- 这个 API 在当前引擎版本里还存在吗？
- 自 LLM 知识截止以来，签名、行为、命名空间变了吗？
- 有已知废弃或新版替代吗？

**三种裁决**：
- `APPROVE`：按描述用，安全
- `CONCERNS [要核实的事项]`：可能有风险，实现前核实
- `REJECT [API 已变 — 提供正确用法]`：API 变了，给正确方案

**类比**：你 2023 年学过 React，2026 年要写 React 代码。这个门禁就是"先查一下 React 现在的文档，你记忆里的 API 还对不对"。

---

### 门禁的三种审查模式（Review Mode）

来自 [director-gates.md#L26](file:///workspace/.claude/docs/director-gates.md)：

| 模式 | 跑哪些门禁 | 适合 |
|------|-----------|------|
| `full` | 全部门禁都跑 | 团队、学习者、要彻底反馈 |
| `lean` | **只跑 PHASE-GATE**（阶段门禁），跳过技能内门禁 | **默认**。单人开发、小团队 |
| `solo` | 全部跳过 | Game Jam、原型、最大速度 |

**配置位置**：[production/review-mode.txt](file:///workspace/production/review-mode.txt)，一个单词：`full` / `lean` / `solo`。

**单次覆盖**：任何技能支持 `--review [full|lean|solo]` 参数，只影响这次运行。

**为什么默认 lean**：单人开发者不需要每写一个 ADR 就跑 4 个总监。只在阶段切换时（`/gate-check`）跑一次就够。

## 第四类：其他高频缩写

### Control Manifest（控制清单）

**是什么**：从所有 `Accepted` ADR 提炼出的"程序员速查表"——**什么必须做、什么绝不能做**，按系统按层组织。

**放哪**：`docs/architecture/control-manifest.md`

**和 ADR 的区别**：
- ADR 解释 **why**（为什么这么决策）
- Control Manifest 告诉你 **what**（具体做什么/不做什么）

**生成命令**：`/create-control-manifest`

---

### Layer（层）

本项目把代码分成几个层，每个层有不同的规则。故事卡里的 `Layer` 字段决定派哪个程序员。

常见层：`Foundation`（基础层）、`Core`（核心层）、`Gameplay`（玩法层）、`UI`、`Audio`、`VFX` 等。

---

### MDA Framework

**是什么**：一种游戏设计分析框架——Mechanics（机制）/ Dynamics（动态）/ Aesthetics（美学/体验）。

**用在哪**：`/brainstorm` 和 `/design-system` 用 MDA 拆解游戏概念。

- **Mechanics**：规则、数据、算法（命中判定公式）
- **Dynamics**：机制运行时产生的行为（玩家发现连招）
- **Aesthetics**：玩家体验到的情感（紧张、爽快）

---

### MVP — Minimum Viable Product（最小可行产品）

**是什么**：最简版本的游戏，包含核心玩法，用于验证可行性。

**本项目用法**：Systems Design 阶段要挑出 MVP 系统，先设计先实现这些。

---

### Sprint（冲刺）

**是什么**：敏捷开发的时间盒，通常 1-2 周。一个冲刺完成一批故事卡。

**相关门禁**：`PR-SPRINT`（冲刺开始前评估可行性）、`QL-TEST-COVERAGE`（冲刺结束时查测试覆盖率）。

---

### Out of Scope（范围外）

**是什么**：故事卡里明确写"不做的事"。防止范围蔓延。

**为什么重要**：程序员看到这个故事卡时，知道哪些不该碰。防止"顺手改了别的"导致越权。

---


## 小测验

**Q1**：`TD-ENGINE-RISK` 是谁的门禁？查什么？
> **A**：技术总监（TD = Technical Director）的门禁。查架构决策用到的引擎 API 是否在 LLM 知识截止后发生了变化，防止写过时代码。

**Q2**：ADR 的 4 种状态是什么？"ADR Approve" 指哪种？
> **A**：`Proposed`（已提议）、`Accepted`（已接受）、`Deprecated`（已废弃）、`Superseded by ADR-NNNN`（被取代）。"ADR Approve" 指从 `Proposed` 变为 `Accepted`，只有 `Accepted` 的 ADR 才能指导编码。

**Q3**：`LP-` 和 `QL-` 开头的门禁是谁的？属于哪一层？
> **A**：`LP-` 是主程序员（Lead Programmer），`QL-` 是 QA 主管（QA Lead）。都属于 Tier 2 Lead 层，管执行细节（代码审查、测试覆盖），区别于 Tier 1 总监层（管宏观）。

**Q4**：门禁的三种裁决是什么？一个 NOT READY 会怎样？
> **A**：`APPROVE/READY`（通过）、`CONCERNS`（有顾虑但可推进）、`REJECT/NOT READY`（不通过）。阶段门禁取最严——一个 NOT READY 整个阶段切换 FAIL。

**Q5**：`full` / `lean` / `solo` 三种审查模式有什么区别？默认哪种？
> **A**：`full` 跑全部门禁；`lean` 只跑 PHASE-GATE（阶段门禁），跳过技能内门禁；`solo` 全部跳过。**默认 lean**，适合单人开发。

**Q6**：GDD、TR、ADR、Story 是什么关系？
> **A**：GDD（设计文档）提炼出 TR（技术需求），TR 被 ADR（架构决策）满足，ADR 被拆成 Story（故事卡）实现。一条链：设计 → 技术指标 → 架构方案 → 任务卡。
