# 12 · 从零编排智能体协作流程

前面 11 篇讲了"零件是什么"。这一篇是**综合实操**——回答四个问题：

1. 主会话是怎么触发智能体调用的？
2. 智能体触发规则写在哪里？
3. 一个完整的智能体调用流程长什么样？
4. 这个工作室从零开始怎么搭起来？

读完这篇，你应该能拿着任何一个空项目，照着步骤把一套多智能体协作编排搭出来。

---

## 第一部分：原理篇 —— 主会话如何触发智能体

### 1.1 触发的本质：`Task` 工具 + `subagent_type`

Claude Code 的主会话（你在终端里敲 `claude` 开的那个会话）本身就是一个 AI。它要"变成一支团队"，靠的不是魔法，而是一个内置工具：**`Task`**。

`Task` 工具的作用是：**在当前会话里派生一个子智能体**。子智能体在自己的上下文窗口里独立运行，干完活把结果返回给主会话。

最简的调用长这样：

```
Task(
  subagent_type: "gameplay-programmer",
  description: "实现近战命中检测",
  prompt: "读 production/epics/combat/melee-hitbox.md，按 ADR 实现到 src/gameplay/combat/，写测试到 tests/combat/"
)
```

- `subagent_type` —— 指定派哪个智能体。这个名字必须对应 [.claude/agents/](file:///workspace/.claude/agents/) 下一个 `[name].md` 文件。
- `prompt` —— 给子智能体的任务说明。
- 子智能体跑完后，返回一段文本结果给主会话。

**类比**：主会话是"项目经理"，`Task` 是它"派活"的动作。它喊一声"gameplay-programmer，把这个故事实现了"，gameplay-programmer 就在自己的工位（独立上下文）干完活，把成果交回来。

**关键点**：主会话**不会自动**派智能体。它什么时候派、派谁、派几个，全靠**技能（skill）的正文**告诉它。这就是下一节要讲的"触发规则"。

---

### 1.2 触发规则写在哪里：三层规则源

智能体的触发规则分散在三层文件里，各管一摊：

```
┌─────────────────────────────────────────────────────────┐
│  第 1 层：智能体定义文件（"员工档案"）                    │
│  .claude/agents/[name].md                                │
│  → 定义：这个智能体是谁、能用什么工具、用什么模型        │
├─────────────────────────────────────────────────────────┤
│  第 2 层：技能编排文件（"作业指导书"）                   │
│  .claude/skills/[name]/SKILL.md                          │
│  → 定义：什么时候派哪个智能体、按什么顺序、并行还是串行  │
├─────────────────────────────────────────────────────────┤
│  第 3 层：协调规则文档（"公司制度")                      │
│  .claude/docs/coordination-rules.md                      │
│  .claude/docs/agent-coordination-map.md                 │
│  .claude/docs/director-gates.md                         │
│  → 定义：谁能委派给谁、冲突怎么升级、门禁怎么过         │
└─────────────────────────────────────────────────────────┘
```

#### 第 1 层：智能体定义（员工档案）

位置：[.claude/agents/](file:///workspace/.claude/agents/)，每个智能体一个 `.md` 文件。

看 [creative-director.md](file:///workspace/.claude/agents/creative-director.md) 的开头：

```yaml
---
name: creative-director
description: "The Creative Director is the highest-level creative authority..."
tools: Read, Glob, Grep, Write, Edit, WebSearch
model: opus
maxTurns: 30
disallowedTools: Bash
skills: [brainstorm, design-review]
---
```

**这里定义的是"这个智能体长什么样"**：

| 字段 | 作用 | 编排意义 |
|------|------|----------|
| `name` | 智能体名（= `subagent_type` 的值） | 主会话靠这个名字派它 |
| `tools` | 能用哪些工具 | **最小权限原则**——总监禁 `Bash`，QA 用 Haiku |
| `model` | 用哪个模型 | 总监用 Opus（贵但聪明），格式化任务用 Haiku（便宜） |
| `maxTurns` | 最多跑几轮 | 防止子智能体跑飞 |
| `disallowedTools` | 显式禁用的工具 | 防止越权（总监不该跑命令） |

**这一层不管"什么时候派"，只管"派出来之后它是什么样"**。

#### 第 2 层：技能编排（作业指导书）

位置：[.claude/skills/](file:///workspace/.claude/skills/)，每个技能一个子目录，里面有 `SKILL.md`。

**这是触发规则的核心**。主会话什么时候派智能体、派谁、派几个、按什么顺序，全写在 skill 的正文里。

看 [team-combat/SKILL.md](file:///workspace/.claude/skills/team-combat/SKILL.md) 的编排指令：

```markdown
## How to Delegate

Use the Task tool to spawn each team member as a subagent:
- subagent_type: game-designer — Design the mechanic
- subagent_type: gameplay-programmer — Implement the core gameplay code
- subagent_type: ai-programmer — Implement NPC/enemy AI behavior
- subagent_type: technical-artist — Create VFX, shader effects
- subagent_type: sound-designer — Define audio events
- subagent_type: [primary engine specialist] — Engine idiom validation
- subagent_type: qa-tester — Write test cases

## Pipeline
### Phase 1: Design → Delegate to game-designer
### Phase 2: Architecture → Delegate to gameplay-programmer (+ ai-programmer)
### Phase 3: Implementation (parallel where possible)
  - gameplay-programmer: Implement core combat code
  - ai-programmer: Implement AI behaviors
  - technical-artist: Create VFX
  - sound-designer: Define audio event list
### Phase 4: Integration
### Phase 5: Validation → Delegate to qa-tester
```

**这就是触发规则**。它告诉主会话：

- **派谁**：7 个智能体（用 `subagent_type` 列出来）
- **顺序**：Phase 1 → 2 → 3 → 4 → 5
- **并行还是串行**：Phase 3 标了 "parallel where possible"——4 个独立的实现智能体一起派
- **决策点**：每个 Phase 之间用 `AskUserQuestion` 让用户确认

**关键设计**：skill 用**自然语言**写编排逻辑，AI 读了之后照着执行。不是代码，是"给 AI 看的 SOP"。

#### 第 3 层：协调规则（公司制度）

位置：[.claude/docs/](file:///workspace/.claude/docs/)。

这一层管"跨技能的通用规则"——单个 skill 管不到的全局纪律。

| 文件 | 管什么 | 关键规则 |
|------|--------|----------|
| [coordination-rules.md](file:///workspace/.claude/docs/coordination-rules.md) | 委派/并行/冲突 | 垂直委派不跳层、同级可咨询不能下命令、冲突找共同上级 |
| [agent-coordination-map.md](file:///workspace/.claude/docs/agent-coordination-map.md) | 谁能指挥谁 | 委派关系表、升级路径表、9 种工作流模式 |
| [director-gates.md](file:///workspace/.claude/docs/director-gates.md) | 总监门禁 | 20+ 个门禁 ID（CD-PILLARS、TD-ARCHITECTURE 等）、三态裁决（APPROVE/CONCERNS/REJECT） |
| [workflow-catalog.yaml](file:///workspace/.claude/docs/workflow-catalog.yaml) | 7 阶段流水线 | Concept → Systems Design → Technical Setup → Pre-Production → Production → Polish → Release |

**这一层是"宪法"**。任何 skill 在派智能体时，都要遵守这里的规则（比如不能跳层委派、跨域改动要 producer 协调）。

#### 三层的关系

```
用户敲 /team-combat
       │
       ▼
主会话读 team-combat/SKILL.md（第 2 层：作业指导书）
       │
       │  SKILL.md 说"派 game-designer"
       ▼
主会话读 .claude/agents/game-designer.md（第 1 层：员工档案）
       │  → 知道它用 Sonnet、能用 Read/Write/Task、maxTurns 20
       │
       │  SKILL.md 说"并行派 4 个实现智能体"
       ▼
主会话查 coordination-rules.md（第 3 层：公司制度）
       │  → 确认并行协议：先发后等、收齐再走、阻塞立即上报
       ▼
主会话发 4 个 Task 调用（并行）
```

### 1.3 三种触发路径

智能体被派生有三种典型路径：

#### 路径 A：用户直接调用单智能体 skill

```
用户敲 /dev-story path
  → skill 内部路由表决定派哪个 programmer
  → Task 派 1 个子智能体
```

**特征**：单智能体、串行、用户驱动。最高频的路径。

#### 路径 B：编排 skill 派生多智能体

```
用户敲 /team-combat "melee parry system"
  → skill 按 Phase 1→2→3→4→5 编排
  → Phase 3 并行派 4 个子智能体
  → 每个 Phase 之间 AskUserQuestion 让用户确认
```

**特征**：多智能体、有并行有串行、人在环中决策。复杂功能的路径。

#### 路径 C：总监门禁并行派生

```
用户敲 /gate-check production
  → skill 并行派 4 个总监（creative/technical/producer/art-director）
  → 每个总监跑各自的 PHASE-GATE
  → 收齐 4 个裁决，取最严格的（一个 NOT READY 全部 FAIL）
```

**特征**：纯并行、裁决式、取最严。阶段切换的把关路径。来自 [director-gates.md](file:///workspace/.claude/docs/director-gates.md) 的 "Parallel Gate Protocol"。

## 第二部分：流程篇 —— 一个完整的智能体调用流程

以最高频的 `/dev-story` 为例，画出从用户敲命令到子智能体返回的**完整调用链**。

### 2.1 调用流程图

```
用户敲: /dev-story production/epics/combat/melee-hitbox.md
  │
  ▼
[1] Claude Code 识别 slash 命令
  │   → 加载 .claude/skills/dev-story/SKILL.md 到主会话上下文
  │
  ▼
[2] Phase 1: 找故事
  │   → Read 故事文件 production/epics/combat/melee-hitbox.md
  │   → 提取: 标题、ID、层、类型、TR-ID、ADR、验收标准、依赖
  │
  ▼

  │   → Read .claude/docs/technical-preferences.md      (命名规范)

  │   依赖没完成? → AskUserQuestion → 用户选 A/B/C

  │   → 查路由表: Core 层 + Logic 类型 + 玩法机制 → gameplay-programmer

  │
  ▼
[5] Phase 4: 派生子智能体（关键步骤）
  │   主会话调用 Task 工具:
  │   ┌─────────────────────────────────────────────┐
  │   │ Task(                                         │
  │   │   subagent_type: "gameplay-programmer",       │
  │   │   prompt: "1. 读故事文件 [path]               │
  │   │            2. 查 TR-ID 在 tr-registry.yaml    │
  │   │            3. 读 ADR 的 Decision + Guidelines  │
  │   │            4. 读 control-manifest 的 Core 层  │
  │   │            5. 读 technical-preferences        │
  │   │            6. 创建测试文件 [path]              │
  │   │            7. 按 ADR 实现，不越 Out of Scope"  │
  │   │ )                                            │
  │   └─────────────────────────────────────────────┘
  │
  ▼
[6] Hook 触发: SubagentStart
  │   → .claude/hooks/log-agent.sh 运行
  │   → 记录到 production/session-logs/agent-audit.log:
  │     "20260622_143022 | Agent invoked: gameplay-programmer"
  │
  ▼
[7] 子智能体在独立上下文里工作
  │   → 读它被告知要读的文件
  │   → 按 ADR 实现指南写代码到 src/gameplay/combat/melee_hitbox.gd
  │   → 写测试到 tests/combat/melee_hitbox_test.gd

  │   → 写完返回一段总结文本
  │
  ▼

  │   → .claude/hooks/log-agent-stop.sh 运行
  │   → 完成审计记录
  │
  ▼
[9] 主会话收到子智能体的返回结果
  │   → Phase 6: 汇总（改了哪些文件、测试写了几个、有无偏差）
  │   → 输出实现摘要给用户
  │
  ▼
[10] Phase 7: 更新会话状态
  │   → 追加到 production/session-state/active.md
  │   → "Story: melee-hitbox — Files changed: [list] — Next: /code-review"
  │
  ▼
[11] 推荐下一步
      → "Ready for: /code-review [files] then /story-done [story-path]"
```

### 2.2 关键节点详解

#### 节点 [3]：为什么并行读 5 个文件？

来自 [dev-story/SKILL.md](file:///workspace/.claude/skills/dev-story/SKILL.md) Phase 2：

> Read all of the following simultaneously — these are independent reads. Do not start implementation until all context is loaded.

**编排智慧**：这 5 个文件互不依赖（读 ADR 不需要先读 TR registry），所以一起发、一起等。如果串行读，要等 5 倍时间。这是 [coordination-rules.md](file:///workspace/.claude/docs/coordination-rules.md) "Parallel Task Protocol" 的体现：**先发后等，收齐再走**。

#### 节点 [5]：为什么给路径不给内容？

skill 告诉子智能体"读 [path]"，而不是把文件内容塞进 prompt。原因：

- **省 token**：主会话不复制大段内容到子智能体的 prompt
- **子智能体自己读最新版**：避免主会话读了之后文件被改，子智能体拿到旧内容
- **职责清晰**：子智能体自己决定读哪几行，主会话不替它做决定

#### 节点 [6][8]：Hook 在流程里的角色

Hook 不是"触发智能体"的机制，而是"智能体被触发时自动记录"的机制。

来自 [.claude/settings.json](file:///workspace/.claude/settings.json)：

```json
"SubagentStart": [{ "matcher": "", "hooks": [{ "command": "bash .claude/hooks/log-agent.sh" }] }],
"SubagentStop":  [{ "matcher": "", "hooks": [{ "command": "bash .claude/hooks/log-agent-stop.sh" }] }]
```

每次主会话派生子智能体，`log-agent.sh` 自动记一行审计日志。这是"无侵入观测"——编排逻辑不用改，hook 自动记录谁被派了、什么时候派的、派了多久。

#### 节点 [9]：子智能体返回什么？

子智能体不直接修改主会话的上下文。它返回一段**文本结果**给主会话，主会话再基于这段文本决定下一步。

**这是编排的关键约束**：子智能体的"成果"是它写的文件（代码、测试、文档），不是它返回的文本。返回的文本只是"汇报"。主会话要汇总、要决策、要推荐下一步，都基于这段汇报 + 它自己再读一遍文件。

### 2.3 三种触发模式的对比

| 模式 | 触发命令 | 派几个 | 并行/串行 | 人在环 |
|------|----------|--------|-----------|--------|
| 单智能体 | `/dev-story` | 1 | 串行 | 派之前确认上下文 |
| 编排团队 | `/team-combat` | 4-7 | 混合（Phase 3 并行） | 每个 Phase 之间确认 |
| 总监门禁 | `/gate-check` | 4 | 纯并行 | 收齐裁决后用户决定是否推进 |

## 第三部分：搭建篇 —— 工作室如何从零建立

这一部分是**实操指南**。假设你拿到一个空项目，要照着这个范例搭一套多智能体协作编排，按 8 步走。

### 步骤 1：写 `CLAUDE.md`（定方向）

**目的**：告诉 AI 这个项目是什么、技术栈是什么、要遵守哪些总原则。

**最小可用版本**：

```markdown
# 我的项目 — 多智能体编排示范

## Technology Stack
- **语言**: Python 3.11
- **框架**: FastAPI
- **版本控制**: Git

## Coordination Rules
@.claude/docs/coordination-rules.md

## Collaboration Protocol
**User-driven collaboration, not autonomous execution.**
Every task follows: Question -> Options -> Decision -> Draft -> Approval
```

**关键点**：
- 用 `@路径` 引用子文件，保持 `CLAUDE.md` 精简（参考本项目的 [CLAUDE.md](file:///workspace/CLAUDE.md) 只有 60 行）
- 写明"人在环"协议——这是防止 AI 越权的总开关
- 技术栈要写清楚，智能体会读这个决定用什么 API

**类比**：公司章程。开篇定调子，细节引用到各部门制度。

---

### 步骤 2：创建 `.claude/` 目录骨架

**目的**：搭好"配置零件"的容器。

```
.claude/
  settings.json          # 机器配置（权限+钩子）
  agents/                # 智能体定义（员工档案）
  skills/                # 技能定义（作业指导书）
  hooks/                 # 钩子脚本（自动守卫）
  rules/                 # 路径规则（按目录生效）
  docs/                  # 协调规则文档（公司制度）
    coordination-rules.md
    agent-coordination-map.md
    director-gates.md
    workflow-catalog.yaml
```

**关键点**：先建空目录，后面逐步填内容。不要一上来就写 49 个智能体——先跑通一个，再扩展。

---

### 步骤 3：定义第一个智能体（`.claude/agents/`）

**目的**：先定义一个最小可用的智能体，验证 `Task` 工具能派生它。

**最小可用智能体**（参考 [gameplay-programmer.md](file:///workspace/.claude/agents/gameplay-programmer.md)）：

```markdown
---
name: backend-programmer
description: "后端程序员，实现 API 和业务逻辑代码"
tools: Read, Glob, Grep, Write, Edit, Bash
model: sonnet
maxTurns: 20
---

You are a Backend Programmer. You translate design documents into clean,
tested code following the project's architecture decisions.

### Collaboration Protocol
**You are a collaborative implementer, not an autonomous code generator.**
1. Read the design document
2. Ask architecture questions if ambiguous
3. Propose approach before implementing
4. Get approval before writing files: "May I write this to [path]?"
5. Implement with transparency
6. Offer next steps

### What This Agent Must NOT Do
- Make architectural decisions (delegate to tech-lead)
- Modify files outside src/api/ without delegation
- Skip tests for logic stories
```

**关键设计**：
- `tools` 用**最小权限**——只给真正需要的工具
- `maxTurns` 设上限（20）防止跑飞
- 正文写"协作协议"——明确它是"提议者"不是"决定者"
- 写"Must NOT Do"——防止越权

**验证**：在主会话里手动派一次，确认能跑通：

```
Task(subagent_type: "backend-programmer", prompt: "读 README.md，总结项目结构")
```

---

### 步骤 4：定义第一个技能（`.claude/skills/`）

**目的**：把"派智能体"的逻辑固化成可复用的 `/命令`。

**最小可用技能**（参考 [dev-story/SKILL.md](file:///workspace/.claude/skills/dev-story/SKILL.md)）：

```markdown
---
name: implement-story
description: "读一个故事文件，路由到正确的程序员，实现它"
argument-hint: "[story-path]"
user-invocable: true
allowed-tools: Read, Glob, Grep, Write, Bash, Task, AskUserQuestion
model: sonnet
---

# Implement Story

## Phase 1: 找故事
- 如果有参数: Read 那个文件
- 如果没参数: Glob production/stories/**/*.md，列出 Status: Ready 的

## Phase 2: 加载上下文
并行读:
- 故事文件（提取验收标准、依赖、Out of Scope）
- 治理 ADR（如果故事里写了 ADR 路径）
- 技术偏好（.claude/docs/technical-preferences.md）

依赖没完成? → AskUserQuestion 让用户选 A/B/C

## Phase 3: 路由
| 故事类型 | 派给 |
|----------|------|
| API 逻辑 | backend-programmer |
| 数据库迁移 | db-engineer |
| 前端组件 | frontend-programmer |

## Phase 4: 实现
用 Task 工具派生子智能体:
Task(
  subagent_type: "[路由结果]",
  prompt: "1. 读故事 [path]  2. 读 ADR [path]  3. 按 ADR 实现  4. 写测试"
)

## Phase 5: 汇总
- 收集子智能体返回的结果
- 输出: 改了哪些文件、测试写了几个、有无偏差
- 推荐: /code-review [files] 然后 /story-done [path]
```

**关键设计**：
- `allowed-tools` 必须包含 `Task`——否则 skill 不能派子智能体
- `user-invocable: true`——用户能直接敲 `/implement-story` 触发
- 正文分 Phase——给 AI 读的 SOP，每步做什么写清楚
- 路由表——明确什么情况派谁，不让 AI 猜

**验证**：敲 `/implement-story production/stories/xxx.md`，看主会话是否按 Phase 走、是否派对了智能体。

### 步骤 5：配置 `settings.json`（钩子 + 权限）

**目的**：搭好"门禁系统"——自动放行安全操作、拦住危险操作、在关键事件自动校验。

**最小可用配置**（参考 [.claude/settings.json](file:///workspace/.claude/settings.json)）：

```json
{
  "permissions": {
    "allow": [
      "Bash(git status*)", "Bash(git diff*)", "Bash(git log*)",
      "Bash(pytest*)"
    ],
    "deny": [
      "Bash(rm -rf *)", "Bash(git push --force*)",
      "Bash(git reset --hard*)", "Read(**/.env*)"
    ]
  },
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash", "hooks": [
        { "type": "command", "command": "bash .claude/hooks/validate-commit.sh", "timeout": 15 }
      ]}
    ],
    "SubagentStart": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "bash .claude/hooks/log-agent.sh", "timeout": 5 }
      ]}
    ],
    "SubagentStop": [
      { "matcher": "", "hooks": [
        { "type": "command", "command": "bash .claude/hooks/log-agent-stop.sh", "timeout": 5 }
      ]}
    ]
  }
}
```

**关键设计**：
- `permissions.allow`——白名单自动放行（git status、测试），减少用户确认摩擦
- `permissions.deny`——黑名单永远拦（`rm -rf`、强推、读 `.env`）
- `SubagentStart/Stop` hook——自动记录审计日志，无侵入观测智能体调用

**类比**：门禁系统。白名单的进，黑名单的拦，特定时刻自动巡检。

### 步骤 6：写协调规则（`.claude/docs/`）

**目的**：定"公司制度"——跨技能通用的纪律。

**最小规则集**（参考 [coordination-rules.md](file:///workspace/.claude/docs/coordination-rules.md)）：

```markdown
# Agent Coordination Rules

1. **Vertical Delegation**: 主管委派给专家，不跳层。
2. **Horizontal Consultation**: 同级可咨询，不能下命令。
3. **Conflict Resolution**: 冲突找共同上级。
4. **Change Propagation**: 跨域改动由总协调者协调。
5. **No Unilateral Cross-Domain Changes**: 不许改别人家的文件。

## Parallel Task Protocol
1. 先发后等：所有独立的 Task 调用一起发出
2. 收齐再走：所有结果收齐再进依赖阶段
3. 阻塞立即上报：任何智能体 BLOCKED 立刻告诉用户
4. 必出部分报告：哪怕部分阻塞也要汇报已完成的部分
```

**委派关系表**（参考 [agent-coordination-map.md](file:///workspace/.claude/docs/agent-coordination-map.md)）：

```markdown
## Delegation Rules
| 谁委派 | 能委派给 |
|--------|----------|
| tech-lead | backend-programmer, frontend-programmer, db-engineer |
| product-manager | 任何智能体（只能在对方领域内） |

## Escalation Paths
| 冲突场景 | 升级给谁 |
|----------|----------|
| 两个程序员对实现有分歧 | tech-lead |
| 产品 vs 技术可行性 | product-manager 协调 |
```

**关键点**：规则要**少而硬**。5 条核心规则 + 1 张委派表 + 1 张升级表，足够起步。规则越多，AI 越容易绕过。

### 步骤 7：跑通第一个端到端流程

**目的**：验证整条链路能走通——从用户敲命令到子智能体返回。

**测试用例**：

1. 创建一个最小故事文件 `production/stories/test-001.md`：
   ```markdown
   # Story: 实现健康检查接口
   - Type: Logic
   - Layer: Core
   - Acceptance Criteria:
     - [ ] GET /health 返回 200
     - [ ] 响应体包含 {"status": "ok"}
   ```

2. 敲 `/implement-story production/stories/test-001.md`

3. 验证：
   - 主会话是否按 Phase 1→2→3→4→5 走？
   - 是否派了 `backend-programmer`？
   - `agent-audit.log` 是否有记录？
   - 子智能体是否遵守"May I write to [path]?"协议？
   - 是否输出了汇总和下一步推荐？

**跑不通的常见原因**：
- skill 的 `allowed-tools` 没包含 `Task` → 不能派子智能体
- 智能体名拼错 → `subagent_type` 找不到对应文件
- `maxTurns` 太小 → 子智能体没干完就停了
- 没写协作协议 → 子智能体直接写文件不问用户

### 步骤 8：扩展到多智能体协作

**目的**：从单智能体扩展到团队编排。

**扩展顺序**（参考本项目的演化路径）：

1. **加第二个智能体**：比如 `qa-tester`，让 `/implement-story` 完成后能派 QA 写测试
2. **加编排 skill**：参考 [team-combat/SKILL.md](file:///workspace/.claude/skills/team-combat/SKILL.md) 写一个 `/team-feature`，编排多个智能体并行
3. **加总监层**：加一个 `tech-lead`（Opus 模型），让它做架构决策和代码审查
4. **加门禁**：参考 [director-gates.md](file:///workspace/.claude/docs/director-gates.md) 定义 `TL-ARCHITECTURE`、`TL-CODE-REVIEW` 等门禁 ID
5. **加阶段流水线**：参考 [workflow-catalog.yaml](file:///workspace/.claude/docs/workflow-catalog.yaml) 定义 Concept → ... → Release 的 7 阶段
6. **加路径规则**：参考 [.claude/rules/](file:///workspace/.claude/rules/) 给 `src/api/`、`src/db/` 等目录写专属规则

**关键原则**：**每加一层，先验证不破坏已有流程**。不要一次性加 10 个智能体——加一个、跑通、再加下一个。

## 第四部分：编排篇 —— 如何从零设计协作流程

这一部分回答最核心的问题：**拿到一个业务需求，怎么设计智能体协作流程？**

### 4.1 编排的五个核心问题

设计任何协作流程前，先回答这五个问题：

| 问题 | 回答了什么 | 写在哪里 |
|------|-----------|----------|
| **谁干？** | 派哪些智能体 | skill 的 "Team Composition" 段 |
| **什么顺序？** | 串行还是并行、谁先谁后 | skill 的 "Pipeline" 段 |
| **什么时候问用户？** | 哪些决策点要人在环 | skill 里 `AskUserQuestion` 的位置 |
| **出错怎么办？** | 子智能体 BLOCKED 的处理 | skill 的 "Error Recovery Protocol" |
| **怎么收尾？** | 汇总什么、推荐什么下一步 | skill 的 "Output" 和 "Next Steps" |

**这五个问题答完，协作流程就成型了**。本项目的所有 `team-*` skill 都按这个结构写。

---

### 4.2 编排模式库（从本项目提炼）

本项目示范了 9 种可复用的编排模式，全在 [agent-coordination-map.md](file:///workspace/.claude/docs/agent-coordination-map.md) 的 "Common Workflow Patterns" 里。提炼成模式库：

#### 模式 1：单智能体实现（最简）

```
用户 → skill → 路由表 → Task(1 个智能体) → 汇总
```
- **适用**：明确单一职责的任务（实现一个故事、写一个测试）
- **范例**：`/dev-story`

#### 模式 2：流水线编排（串行多阶段）

```
Phase 1 → Phase 2 → Phase 3 → ... → Phase N
每个 Phase 派 1 个智能体，串行
```
- **适用**：有明确阶段依赖的流程（设计→架构→实现→测试）
- **范例**：`/team-combat` 的 Phase 1→2→4→5

#### 模式 3：并行扇出（同阶段多智能体）

```
       ┌→ Agent A ┐
Phase →├→ Agent B ├→ 汇总
       └→ Agent C ┘
```
- **适用**：同一阶段多个互不依赖的子任务
- **范例**：`/team-combat` Phase 3（gameplay + ai + vfx + audio 并行）

#### 模式 4：门禁并行裁决（取最严）

```
       ┌→ Director 1 → APPROVE    ┐
gate →├→ Director 2 → CONCERNS    ├→ 取最严 → PASS/FAIL
       ├→ Director 3 → NOT READY  ┤
       └→ Director 4 → APPROVE    ┘
```
- **适用**：阶段切换前的多维度把关
- **范例**：`/gate-check` 并行派 4 个总监

#### 模式 5：升级链（冲突找上级）

```
Specialist A ⟷ Specialist B（谈不拢）
       ↓ 升级
Lead（仲裁）
       ↓ 还谈不拢
Director（最终裁决）
```
- **适用**：智能体之间有分歧
- **范例**：[agent-coordination-map.md](file:///workspace/.claude/docs/agent-coordination-map.md) 的 "Escalation Paths"

#### 模式 6：变更传播（跨域协调）

```
Design 变更 → 通知 Lead Programmer + QA Lead + Producer
            → Producer 协调跨部门影响
```
- **适用**：一个变更影响多个域
- **范例**：[coordination-rules.md](file:///workspace/.claude/docs/coordination-rules.md) 第 4 条

#### 模式 7：循环迭代（冲刺周期）

```
sprint-plan → [所有智能体执行] → sprint-status → retrospective → 下一个 sprint
```
- **适用**：周期性重复流程
- **范例**：`/sprint-plan` → `/dev-story` ×N → `/retrospective`

#### 模式 8：错误恢复（部分失败）

```
Task 派 4 个智能体 → 1 个 BLOCKED
  → 立即上报用户
  → AskUserQuestion: 跳过/重试/停下
  → 必出部分报告
```
- **适用**：任何并行编排
- **范例**：所有 `team-*` skill 的 "Error Recovery Protocol"

#### 模式 9：人在环决策（每个 Phase 确认）

```
Phase 1 → AskUserQuestion → 用户选 A/B/C → Phase 2 → AskUserQuestion → ...
```
- **适用**：高 stakes 流程，每步都要人确认
- **范例**：`/team-combat` 每个 Phase 之间

### 4.3 编排清单（设计新流程时逐项检查）

设计一个新的协作流程时，照这个清单逐项确认：

- [ ] **角色**：派哪些智能体？每个的 `subagent_type` 对应 `.claude/agents/` 里有文件吗？
- [ ] **顺序**：哪些 Phase 串行、哪些并行？并行的真的互不依赖吗？
- [ ] **上下文**：每个子智能体被告知要读哪些文件？给路径还是给内容？（给路径）
- [ ] **决策点**：哪些地方要 `AskUserQuestion`？选项设计好了吗？
- [ ] **错误恢复**：子智能体 BLOCKED 怎么办？跳过/重试/停下？
- [ ] **汇总**：收齐结果后输出什么格式？改了哪些文件、测试几个、有无偏差？
- [ ] **下一步**：推荐用户接下来跑什么命令？
- [ ] **权限**：skill 的 `allowed-tools` 包含 `Task` 吗？
- [ ] **协调规则**：派智能体时遵守了"不跳层、不跨域"吗？
- [ ] **审计**：`SubagentStart/Stop` hook 会自动记录，不用手动管

## 小测验

**Q1**：主会话派智能体靠什么工具？`subagent_type` 的值对应哪个文件？
> **A**：靠 `Task` 工具。`subagent_type` 的值对应 `.claude/agents/[name].md` 文件。

**Q2**：智能体的"触发规则"主要写在哪一层？
> **A**：第 2 层——`.claude/skills/[name]/SKILL.md` 的正文。它用自然语言告诉主会话"什么时候派谁、按什么顺序"。

**Q3**：为什么 `/dev-story` 的 Phase 2 要并行读 5 个文件而不是串行？
> **A**：这 5 个文件互不依赖（读 ADR 不需要先读 TR registry），并行读省 5 倍时间。这是 [coordination-rules.md](file:///workspace/.claude/docs/coordination-rules.md) "Parallel Task Protocol" 的体现：先发后等，收齐再走。

**Q4**：`SubagentStart` hook 和 `Task` 工具是什么关系？
> **A**：`Task` 是主会话"派"智能体的动作；`SubagentStart` hook 是"智能体被派时自动触发的脚本"。hook 不派智能体，只记录"谁被派了"。这是无侵入观测。

**Q5**：从零搭建工作室，为什么要先跑通单智能体再扩展到多智能体？
> **A**：每加一层都要验证不破坏已有流程。一次性加 10 个智能体，出问题不知道是哪个的锅。增量验证是工程化的基本纪律。

## 一句话总结

> **从零编排智能体协作 = 写 `CLAUDE.md` 定方向 + 写 `agents/` 定角色 + 写 `skills/` 定流程 + 写 `settings.json` 定守卫 + 写 `docs/` 定制度 + 用 `Task` 工具把流程跑起来。**

核心不是"定义很多智能体"，而是"设计好它们怎么协作"。先跑通一个，再扩展到多个，最后加门禁和阶段流水线。
