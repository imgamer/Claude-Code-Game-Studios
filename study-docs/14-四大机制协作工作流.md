# 14 · 四大机制如何协作 —— 一个例子讲透

这一篇回答你的核心问题：

- Agent、skill、门禁、hook 分别在哪个阶段？
- 它们怎么配合？
- 大模型怎么知道要用哪个？靠 description 还是配置？
- 大模型怎么"功能调用"触发这些机制？
- 触发后怎么"返回原来的节点继续流程"？
- 怎么知道执行到哪一步、接着做什么？

**方法**：不抽象讲，用一个真实例子从头到尾走一遍。例子是 `/dev-story` 实现一个故事卡，它同时涉及全部四个机制。

---

## 0. 四大机制一句话定位

| 机制 | 一句话 | 是配置还是描述触发 |
|------|--------|---------------------|
| **Skill** | 流程脚本（"作业指导书"）——告诉主会话按什么 Phase 走、派谁 | 用户敲 `/命令` 显式触发，或主会话根据 `description` 自动选 |
| **Agent** | 角色档案（"员工"）——独立上下文里干活的专家 | skill 正文里用 `Agent`/`Task` 工具 + `subagent_type` 显式派生 |
| **门禁** | 质量关卡（"质检员"）——派总监 agent 给 APPROVE/CONCERNS/REJECT | skill 正文里写"用 gate XX-YY 派 [总监]"显式触发 |
| **Hook** | 自动守卫（"门磁"）——特定事件自动跑脚本 | **纯配置**——在 settings.json 注册，大模型不需要知道 |

**关键区分**：
- skill / agent / 门禁 是**大模型主动调用**的（通过工具调用）
- hook 是**事件自动触发**的（大模型不调用，配置触发）

---

## 1. 例子：实现一个"近战命中检测"故事

用户在终端敲：

```
/dev-story production/epics/combat/melee-hitbox.md
```

这会触发一个完整工作流，涉及全部四个机制。我们逐步看。

---

## 2. 第一步：Skill 如何被触发

### 触发方式 1：显式 slash 命令（本项目主用）

用户敲 `/dev-story`，Claude Code 做两件事：

1. **识别命令**：查 `.claude/skills/` 下有没有 `dev-story` 这个 skill
2. **加载 skill 正文**：把 [.claude/skills/dev-story/SKILL.md](file:///workspace/.claude/skills/dev-story/SKILL.md) 的全部内容注入主会话上下文

这一刻，主会话从一个"通用 AI"变成"正在执行 dev-story 流程的 AI"。skill 的正文就是它的脚本。

### 触发方式 2：自动委派（本项目不用，但官方支持）

如果用户不敲 `/dev-story`，而是说"帮我实现 production/epics/combat/melee-hitbox.md 这个故事"，Claude 会：

1. 扫描所有 skill 的 `description` 字段
2. 看到 dev-story 的描述是 "Read a story file and implement it..."
3. 自动选 dev-story 来处理

**关键**：`description` 写得越清楚，自动匹配越准。本项目选择用 slash 命令显式触发，因为可复现、可审计。

### Skill 的 frontmatter 告诉主会话"我有什么权限"

看 dev-story 的 frontmatter（[SKILL.md#L1-8](file:///workspace/.claude/skills/dev-story/SKILL.md)）：

```yaml
name: dev-story
description: "Read a story file and implement it..."
argument-hint: "[story-path]"
user-invocable: true              # 用户能敲 /dev-story 触发
allowed-tools: Read, Glob, Grep, Write, Bash, Task, AskUserQuestion
model: sonnet
```

**`allowed-tools` 里必须包含 `Task`**——否则这个 skill 不能派 agent。这是配置层面的硬约束：没给 `Task` 权限，就没法派子智能体。

---

## 3. 第二步：Skill 正文驱动主会话按 Phase 走

skill 加载后，主会话按正文里的 Phase 1 → 2 → 3 → 4 → 5 → 6 → 7 走。**"怎么知道执行到哪一步"的答案就在这里**：skill 正文就是脚本，主会话读到哪里就执行到哪里。

```
Phase 1: Find the Story       ← 找故事文件
Phase 2: Load Full Context    ← 加载上下文
Phase 3: Route to Programmer  ← 路由到正确的程序员
Phase 4: Implement            ← 派 agent 实现
Phase 5: Test Evidence        ← 测试要求
Phase 6: Collect and Summarise← 收集汇总
Phase 7: Update Session State ← 更新会话状态
```

**怎么知道当前在哪一步**：
- skill 正文是顺序结构，主会话从上往下执行
- 每个 Phase 有明确的开始和结束标志（下一个 `## Phase N` 就是上一步结束）
- 关键决策点用 `AskUserQuestion` 让用户确认，确认后才进下一步

---

## 4. 第三步：Agent 如何被触发（Phase 4 的核心）

到了 Phase 4，skill 正文写了（[SKILL.md#L177](file:///workspace/.claude/skills/dev-story/SKILL.md)）：

> Spawn the chosen programmer agent(s) via Task with the full context package

主会话读到这里，就**调用 `Agent`/`Task` 工具**派生子智能体。这是大模型的"功能调用"——和它调用 Read、Write 是同一回事。

### 调用过程

主会话生成一个工具调用：

```
Agent(
  subagent_type: "gameplay-programmer",
  description: "实现近战命中检测",
  prompt: "1. 读故事文件 [path]
           2. 查 TR-ID 在 tr-registry.yaml
           3. 读 ADR 的 Decision + Guidelines
           4. 读 control-manifest 的 Core 层
           5. 读 technical-preferences
           6. 创建测试文件 [path]
           7. 按 ADR 实现，不越 Out of Scope"
)
```

**大模型怎么知道派哪个 agent**：
- 不是靠 description 自动匹配（那是自动委派）
- 是 skill 正文 Phase 3 的**路由表**告诉它的（[SKILL.md#L146-154](file:///workspace/.claude/skills/dev-story/SKILL.md)）：

| Story context | Primary agent |
|---|---|
| Foundation layer | `engine-programmer` |
| Type: UI | `ui-programmer` |
| Core — gameplay mechanics | `gameplay-programmer` ← 本例选这个 |
| Core — AI behaviour | `ai-programmer` |

主会话读路由表 + 故事卡的 Layer/Type 字段，决定派 `gameplay-programmer`。**这是 skill 正文显式编排，不是大模型猜的**。

### `subagent_type` 怎么找到 agent

`subagent_type: "gameplay-programmer"` 这个值，对应 [.claude/agents/gameplay-programmer.md](file:///workspace/.claude/agents/gameplay-programmer.md) 文件。Claude Code 启动时扫描 `.claude/agents/` 目录，建一张"name → agent 定义"的表。主会话派生时按名字查表。

## 5. 第四步：Hook 如何被触发（自动，大模型不管）

主会话发出 `Agent` 工具调用的瞬间，Claude Code 触发 `SubagentStart` 事件。这个事件**不是大模型触发的**，是 Claude Code 内部触发的。

### 配置在 settings.json 里

看 [.claude/settings.json#L134-145](file:///workspace/.claude/settings.json)：

```json
"SubagentStart": [
  {
    "matcher": "",
    "hooks": [
      {
        "type": "command",
        "command": "bash .claude/hooks/log-agent.sh",
        "timeout": 5
      }
    ]
  }
]
```

**解读**：
- `matcher: ""` —— 匹配所有子智能体（不筛选类型）
- 每次任何子智能体启动，自动跑 `log-agent.sh`
- timeout 5 秒——超时就杀掉，不阻塞主流程

### hook 脚本干什么

看 [.claude/hooks/log-agent.sh](file:///workspace/.claude/hooks/log-agent.sh)：

```bash
INPUT=$(cat)
AGENT_NAME=$(echo "$INPUT" | jq -r '.agent_type // "unknown"')
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
echo "$TIMESTAMP | Agent invoked: $AGENT_NAME" >> "production/session-logs/agent-audit.log"
exit 0
```

它读 Claude Code 传来的 JSON（含 `agent_type`），往审计日志追加一行。**主会话和子智能体都不知道这件事发生了**——hook 是无侵入的旁路观测。

### hook 的输入从哪来

Claude Code 在触发事件时，把上下文打包成 JSON 通过 stdin 传给 hook 脚本。`SubagentStart` 事件的 JSON 包含 `session_id`、`agent_id`、`agent_type` 等字段。hook 脚本用 jq 解析。

**关键**：hook 的输入是 Claude Code 给的，不是大模型给的。大模型完全不知道 hook 存在。

### 本项目的关键 hook 事件

| 事件 | 什么时候触发 | 配置的 hook | 作用 |
|------|-------------|-------------|------|
| `SessionStart` | 会话启动 | `session-start.sh`、`detect-gaps.sh` | 初始化、检测配置缺口 |
| `PreToolUse` (Bash) | 执行 Bash 命令前 | `validate-commit.sh`、`validate-push.sh` | 拦截危险 git 操作 |
| `PostToolUse` (Write\|Edit) | 写文件后 | `validate-assets.sh`、`validate-skill-change.sh` | 校验资产、防止误改 skill |
| `SubagentStart` | 子智能体启动 | `log-agent.sh` | 记审计日志 |
| `SubagentStop` | 子智能体停止 | `log-agent-stop.sh` | 记完成日志 |
| `PreCompact`/`PostCompact` | 上下文压缩前后 | `pre-compact.sh`、`post-compact.sh` | 压缩前后保存/恢复状态 |
| `Stop` | 会话结束 | `session-stop.sh` | 收尾 |

## 6. 第五步：子智能体干活并返回

子智能体 `gameplay-programmer` 在自己的独立上下文里：
1. 读故事文件、ADR、control-manifest（按 prompt 里给的路径）
2. 写代码到 `src/gameplay/combat/melee_hitbox.gd`
3. 写测试到 `tests/combat/melee_hitbox_test.gd`
4. 返回一段文本总结给主会话

子智能体停止的瞬间，Claude Code 触发 `SubagentStop` 事件 → 跑 `log-agent-stop.sh` 记日志。

### "触发后如何返回原来的触发节点继续流程"

**这是关键问题**。答案：

1. 主会话发出 `Agent` 工具调用后**阻塞等待**（默认前台模式）
2. 子智能体在独立上下文里跑完，返回一段文本
3. Claude Code 把这段文本作为工具调用的"返回值"塞回主会话
4. 主会话的上下文里，`Agent` 工具调用那一行现在有了结果，它接着往下读 skill 正文

**类比**：你（主会话）在写一份报告，写到一半需要查资料。你叫助理（子智能体）去查，助理查完把一张纸条（返回文本）递给你，你接着写报告。助理查资料的过程你不知道也不参与，你只看纸条上的结果。

**主会话怎么"记得"流程到哪了**：因为 skill 正文还在主会话的上下文里。子智能体只是插入了一段"工具调用 → 返回结果"，主会话的上下文没有被覆盖。它读 Phase 4 完成 → 接着读 Phase 5 → 6 → 7。

## 7. 第六步：门禁如何被触发

门禁**不是独立机制**——它是"派总监 agent 跑特定审查"的封装。本例 `/dev-story` 本身不跑门禁，但下一个 skill `/code-review` 会跑 `LP-CODE-REVIEW` 门禁。

我们换到 `/gate-check` 的例子看门禁怎么触发。

### /gate-check 的门禁触发

看 [gate-check/SKILL.md](file:///workspace/.claude/skills/gate-check/SKILL.md)，正文写了：并行派 4 个总监 agent，每个跑自己的 `-PHASE-GATE`。

主会话读到这段，**发 4 个 `Agent` 工具调用（并行）**：

```
Agent(subagent_type: "creative-director", prompt: "跑 CD-PHASE-GATE 门禁，审查 [target phase] 的创意就绪度...")
Agent(subagent_type: "technical-director", prompt: "跑 TD-PHASE-GATE 门禁，审查 [target phase] 的技术就绪度...")
Agent(subagent_type: "producer", prompt: "跑 PR-PHASE-GATE 门禁...")
Agent(subagent_type: "art-director", prompt: "跑 AD-PHASE-GATE 门禁...")
```

### 门禁 prompt 从哪来

不是 skill 现编的，是 [.claude/docs/director-gates.md](file:///workspace/.claude/docs/director-gates.md) 定义好的。skill 只写：

> Spawn `creative-director` via Task using gate **CD-PHASE-GATE** from `.claude/docs/director-gates.md`

主会话读 director-gates.md 找到 `CD-PHASE-GATE` 的 prompt 模板，填入上下文变量（target phase、ADR 列表等），作为 `Agent` 工具的 `prompt` 参数发出。

**为什么这样设计**：消除漂移。所有 skill 引用同一份门禁定义，门禁 prompt 要改只改 director-gates.md 一处，所有 skill 自动生效。

### 门禁的三态裁决

每个总监 agent 跑完返回：
- `READY` / `APPROVE` —— 通过
- `CONCERNS [list]` —— 有顾虑但可推进
- `NOT READY` / `REJECT` [blockers] —— 不通过

主会话收齐 4 个裁决，**取最严**：一个 NOT READY 全部 FAIL。

### 门禁审查模式（review mode）控制要不要跑

来自 [director-gates.md#L26](file:///workspace/.claude/docs/director-gates.md)：

| 模式 | 跑哪些门禁 |
|------|-----------|
| `full` | 全部 |
| `lean`（默认） | 只跑 PHASE-GATE，跳过技能内门禁 |
| `solo` | 全部跳过 |

**主会话怎么知道当前模式**：读 [production/review-mode.txt](file:///workspace/production/review-mode.txt)（一个单词）。skill 正文写了"派门禁前先读这个文件解析模式"。

## 9. 四个机制的角色定位（再强调）

| 机制 | 角色 | 触发方 | 触发方式 | 大模型是否主动调用 |
|------|------|--------|----------|---------------------|
| **Skill** | 流程脚本 | 用户敲 `/命令` 或自动委派 | slash 命令 / description 匹配 | 是（读正文执行） |
| **Agent** | 干活的专家 | skill 正文指示 | `Agent`/`Task` 工具调用 | 是（工具调用） |
| **门禁** | 质量审查 | skill 正文指示 | `Agent` 工具调用 + 引用 director-gates.md | 是（工具调用） |
| **Hook** | 自动守卫 | Claude Code 事件 | settings.json 配置 → 事件触发 | **否**（纯配置，大模型不知道） |

**三个大模型主动调用的，一个配置自动触发的**。这是核心区分。

## 10. 关键问题逐一回答

### Q1: 大模型怎么知道要用哪些机制？靠 description 还是配置？

**靠 skill 正文**。机制是这样的：

1. 用户敲 `/dev-story` → skill 正文注入上下文
2. skill 正文 Phase 3 写了"查路由表派 gameplay-programmer"→ 主会话读到这里就派
3. skill 正文 Phase 4 写了"用 Agent 工具派生"→ 主会话读到这里就调用 Agent 工具

**description 的作用**：
- skill 的 description 用于"用户没敲 /命令时自动选哪个 skill"
- agent 的 description 用于"自动委派时匹配哪个 agent"
- 本项目主要用显式 slash 命令 + skill 正文路由表，所以 description 主要起辅助作用

**配置的作用**：
- `allowed-tools: Task` 是硬约束——没这个权限，skill 不能派 agent
- `permissions.allow/deny` 是硬约束——拦住危险操作
- hook 注册是硬约束——事件触发就跑，大模型管不着

### Q2: 大模型如何进行"功能调用"来触发这些机制？

大模型生成**工具调用**。和它调用 Read/Write 是同一回事。

```
主会话输出:
  我需要派 gameplay-programmer 来实现这个故事
  <tool_call name="Agent">
    <parameter name="subagent_type">gameplay-programmer</parameter>
    <parameter name="description">实现近战命中检测</parameter>
    <parameter name="prompt">读故事文件...</parameter>
  </tool_call>
```

Claude Code 接到这个工具调用，派生子智能体。子智能体跑完返回结果，Claude Code 把结果塞回主会话上下文，主会话接着往下读 skill 正文。

### Q3: 触发后如何返回原来的触发节点继续流程？

**主会话上下文没有被覆盖**。整个过程：

1. 主会话上下文里有 skill 正文（从 Phase 1 到 7 全文）
2. 主会话读到 Phase 4，发出 `Agent` 工具调用
3. **主会话阻塞等待**（默认前台模式）
4. 子智能体在独立上下文跑完，返回文本
5. Claude Code 把返回文本作为"工具调用结果"塞进主会话上下文
6. 主会话上下文现在是：[skill 正文] + [Agent 工具调用记录 + 返回结果]
7. 主会话接着读 skill 正文 Phase 5、6、7

**关键**：子智能体的上下文和主会话的上下文是**隔离的两个**。子智能体干活时，主会话的上下文原封不动地等。子智能体返回后，主会话继续。

**如果用 `run_in_background: true`**：主会话不阻塞，可以继续干别的，但要主动用 `/tasks` 查子任务状态。本项目默认前台模式，简单可控。

### Q4: 怎么知道执行到流程的哪一步？接着要做什么？

**三个机制保证**：

1. **skill 正文是顺序结构**：Phase 1 → 2 → 3 → ... → 7，主会话从上往下读，读到哪执行到哪
2. **关键节点用 `AskUserQuestion`**：依赖没完成？让用户选 A/B/C。模式没确定？让用户选。每个决策点用户确认后才进下一步
3. **会话状态文件**：[production/session-state/active.md](file:///workspace/production/session-state/active.md) 记录"当前故事、改了哪些文件、下一步推荐什么"。即使会话中断，下次启动 hook 会读这个文件恢复上下文

**具体到本例**：
- Phase 6 输出里明确写 `Ready for: /code-review [files] then /story-done [story-path]`
- Phase 7 把这段写进 active.md
- 下次会话启动，`session-start.sh` hook 读 active.md，主会话知道"上次到 /dev-story 完成，下一步 /code-review"

### Q5: 它们分别在哪个阶段？

按游戏开发的 7 阶段（Concept → Systems Design → Technical Setup → Pre-Production → Production → Polish → Release）：

| 阶段 | 主要 skill | 派的 agent | 跑的门禁 |
|------|-----------|-----------|----------|
| Concept | `/brainstorm`、`/design-system` | creative-director、art-director | CD-PILLARS、AD-CONCEPT-VISUAL |
| Systems Design | `/design-system`、`/review-all-gdds` | game-designer、systems-designer | CD-GDD-ALIGN、CD-SYSTEMS |
| Technical Setup | `/create-architecture`、`/architecture-decision`、`/architecture-review`、`/create-control-manifest` | technical-director、engine-programmer | TD-ARCHITECTURE、TD-ADR、TD-ENGINE-RISK |
| Pre-Production | `/story-readiness`、`/dev-story` | gameplay-programmer、qa-tester | QL-STORY-READY、PR-EPIC |
| Production | `/dev-story`、`/code-review`、`/story-done` | 各种程序员、lead-programmer | LP-CODE-REVIEW、QL-TEST-COVERAGE |
| Polish | `/team-qa`、`/playtest` | qa-tester、creative-director | QL-TEST-COVERAGE、CD-PLAYTEST |
| Release | `/gate-check release` | 4 大总监并行 | 4 个 -PHASE-GATE |

**Hook 跨所有阶段**：Hook 不属于任何阶段，它是"事件驱动的自动守卫"，只要事件发生就跑（比如任何阶段派 agent 都触发 SubagentStart）。

## 12. 小测验

**Q1**：skill、agent、门禁、hook 哪些是大模型主动调用的？哪些是配置自动触发的？
> **A**：前三个（skill/agent/门禁）是大模型主动调用工具触发的；hook 是 settings.json 配置、事件自动触发的，大模型不知道 hook 存在。

**Q2**：主会话怎么知道派 `gameplay-programmer` 而不是 `ai-programmer`？
> **A**：不是靠 description 自动匹配，是 skill 正文 Phase 3 的路由表告诉它的。故事卡的 Layer=Core + Type=gameplay mechanics → 路由表查到 `gameplay-programmer`。skill 正文显式编排。

**Q3**：子智能体跑完后，主会话怎么"接着"原来的流程？
> **A**：主会话上下文里 skill 正文一直在。`Agent` 工具调用阻塞等待，子智能体返回后结果塞进上下文，主会话接着读 skill 下一行（下一个 Phase）。

**Q4**：`SubagentStart` hook 是谁触发的？大模型怎么调用它？
> **A**：**大模型不调用它**。Claude Code 在派生子智能体时内部触发 `SubagentStart` 事件，按 settings.json 配置自动跑 `log-agent.sh`。大模型完全无感。

**Q5**：门禁的 prompt 从哪来？skill 现编还是统一管理？
> **A**：统一管理在 [.claude/docs/director-gates.md](file:///workspace/.claude/docs/director-gates.md)。skill 只写"用 gate XX-YY 派 [总监]"，主会话去 director-gates.md 找 prompt 模板。这样改门禁只改一处，所有 skill 自动生效。

**Q6**：怎么知道流程执行到哪一步了？
> **A**：三个机制——(1) skill 正文是顺序结构，读到哪执行到哪；(2) 关键节点 AskUserQuestion 让用户确认；(3) 会话状态文件 `production/session-state/active.md` 记录当前进度和下一步推荐，会话重启时 hook 读取恢复。

**Q7**：`description` 字段的作用是什么？本项目主要靠它触发吗？
> **A**：description 用于"自动委派"——用户没敲 /命令时，Claude 根据 description 匹配 skill/agent。**本项目主要靠 slash 命令显式触发 + skill 正文路由表显式编排**，description 起辅助作用。但写清楚 description 有备无患，支持自动委派场景。
